How it works

Describe, drive, deploy.

From English scenario to a standalone Playwright spec in under a minute.

1

You describe

Write the scenario in English: "log in as standard_user, add a backpack to cart, verify the badge shows 1". No locator syntax, no test framework boilerplate.

2

Claude drives

An agent opens a real Chromium via Microsoft's Playwright MCP, clicks through your site, fills forms, handles multi-step flows, recovers from failed clicks — then validates the scenario end-to-end.

3

Spec emerges

A clean TypeScript Playwright spec lands in tests/ — semantic locators, meaningful assertions, no MCP dependencies. Ready to run, edit, version, ship to any CI.

FAQ

Questions & answers

How much does this cost to run?
The CLI, web UI, and GitHub Action are free (MIT). You pay only for Anthropic API tokens when you generate, heal, refine, or triage. Typical per-run cost: $0.02 (Haiku) to $0.25 (Opus). Every command prints its exact cost at the end. Use --max-cost $0.50 to cap spend per run.
Does my code or DOM get sent to Anthropic?
Yes, when you run commands that call Claude. The agent sends the page's ARIA accessibility tree (not full DOM) and screenshots to Anthropic's API. Anthropic's data policy: API traffic is not used for training. If you need full data sovereignty, the open-source CLI can be forked and pointed at a self-hosted model, though current prompts are tuned for Claude Opus 4.7.
Does this work with my existing Playwright setup?
Yes. Generated specs are standard @playwright/test files that run with npx playwright test. No runtime dependency on Promptomate once a test is written. You can edit specs by hand, mix them with manually-written tests, and keep your existing playwright.config.ts. We add lightweight dirs (.promptomate/ for metadata, tests/ for specs — the standard Playwright location).
Can non-engineers actually use this?
For generating and running tests — yes, via the web UI (promptomate serve) or the GitHub Action in their team's CI. Understanding failure triage output is a few minutes of training. Writing custom assertions still benefits from engineer review, but the mechanical parts (describing scenarios, running suites, checking whether something's broken) are open to everyone.
How is this different from Testim or Mabl?
Open source and local-first. Testim/Mabl are great but they're closed-source cloud products at $5k+/month for a small team. Your tests live in their UI, their cloud runs your browsers, their AI is proprietary. Promptomate is MIT code running on your machine with your API key. Your tests are TypeScript you commit to your own repo. Zero vendor lock-in.
What about flaky tests?
promptomate triage classifies failures as real bug / flake / dom drift by analyzing the test code + the actual failure output + a fresh screenshot. --apply automatically retries flakes with a short pause, heals DOM drift by regenerating the broken locators, and stops on real bugs. Your CI PR comment shows exactly what happened per test.
Which models does it support?
Default is Claude Opus 4.7 (best quality). Use --model sonnet for Sonnet 4.6 (~3× cheaper, still excellent for most flows) or --model haiku for Haiku 4.5 (fastest + cheapest, great for simple generation and triage). You can also set PROMPTOMATE_MODEL env var as a default.
Is there a hosted version I can try?
Not yet — the Team cloud tier is in active development (ETA ~6 weeks). For now, the fastest way to try it is npm install -g promptomate followed by promptomate quickstart (2-min interactive setup + a demo run). Or deploy the web UI to Render/Fly.io for your team — we include a one-click Blueprint.
